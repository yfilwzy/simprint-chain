---

## 一、魔改全景

本地相对官方的三项魔改（经全量扫描确认，范围精确）：

| 魔改项 | 代码位置 | 文件数 | 保留策略 |
|--------|---------|--------|---------|
| 自研更新器 | `services/updater/` + `bin/updater.rs` + `infrastructure/updater/` | ~15 | 采纳官方分层 + 移植本地特有逻辑 |

## 二、Updater 保留方案

### 2.1 现状勘误
官方 v0.2.9 起就有自研 `bin/updater.rs`，且无 `tauri-plugin-updater`。本地 updater 是基于官方自研的二次魔改。因此 updater 的"保留"实为**采纳官方 v0.2.26 的完整分层，叠加本地 URL 配置差异**。

### 2.2 官方 updater 分层（v0.2.26）
```
infrastructure/updater/
├── checker.rs      # POST config.updater.check_url 检查更新
├── planner.rs      # plan_updates / 哈希计算
├── downloader.rs   # 带进度的下载
├── verifier.rs     # SHA256 校验
├── installer.rs    # 文件级热替换 + 回滚
├── manifest.rs     # 版本清单
├── service.rs      # 互斥锁、内存缓存、事件 emit
├── state.rs        # OnceCell/Mutex 状态
└── types.rs        # UpdateEvent 枚举、InstallTask、CheckRequest/Response

services/updater/
├── update_service.rs   # 业务编排（check/download/install）
└── types.rs

services/runtime_updater/   # 浏览器内核独立更新
├── service.rs          # 180s 轮询、协议版本比对
└── types.rs

src/bin/updater.rs      # 独立二进制：install / install-package 两种模式
```

### 2.3 本地需保留的差异点
仅 **配置层差异**，指向用户自有更新服务器：

```toml
# config.example.toml（本地占位模板）
[updater]
check_url = "<YOUR_UPDATE_SERVER>/api/v1/versions/check"
latest_json_url = "<YOUR_CDN>/latest.json"
runtime_latest_json_url = "<YOUR_CDN>/simprint-runtime/latest.json"
updater_temp_dir = "updates"
```

本地 `update_service.rs` 的核心逻辑（`ShellExecuteW runas` 提权、`restart_main_app` 重启）与官方趋同，采纳官方实现即可。

### 2.4 updater.exe 与 splashscreen 启动耦合（关键风险点）
`app/splashscreen.rs::init_startup`（L142-208）在启动流程中硬编码调用 updater 检查/下载/安装，并据此决定是否创建主窗口。融合 updater 时**必须同步改这段启动逻辑**，否则会出现"检查更新阻塞启动"。

**处理**：采纳官方 splashscreen.rs，确认其与官方 updater 分层匹配；本地若有启动流程定制，移植到官方骨架上。

## 三、Proxy_Chain 保留方案（核心决策）

### 3.1 语义辨析（推演结论）

经 sequential-thinking 深度推演，本地 proxy_chain 与官方 mihomo 是**不同概念**：

| 维度 | 本地 proxy_chain | 官方 v0.2.22 mihomo |
|------|------------------|---------------------|
| **拓扑** | 多跳串联（前置跳 → 落地） | 单跳（选一个节点） |
| **机制** | 落地 SOCKS5 带 `dialer-proxy` 指向 FIRST_HOP 组 | 连外部 Clash-Verge controller，选节点编译成本地 listener |
| **目的** | 反指纹/隐藏真实出口 IP | 本地代理节点编排 |
| **YAML 生成** | 自行生成完整 mihomo YAML（generator.rs，无依赖） | 依赖外部 controller |
| **进程管理** | 独立启动 mihomo 进程（process.rs） | 连接已运行的 mihomo |

### 3.2 保留方案：方案 A（保留独立模块 + 前端分层）

**理由**：
1. 功能零损失——proxy_chain 的多跳链式是本地核心价值
2. 数据隔离——proxy_chain 自行生成 YAML + 独立进程，与官方 mihomo controller 不冲突
3. 概念清晰——前端分层让用户明确区分"链式代理(多跳)"和"节点代理(官方单跳)"

### 3.3 前端分层设计

`plugins/pages/proxy-center/` 顶层 Tab 分层：
```
代理中心
├── Tab 1: 链式代理（本地多跳，proxy_chain）
│   └── ProxyChainPanel（本地原有组件）
│       ├── 订阅管理
│       ├── 节点测速
│       ├── 策略选择（Select/Fallback/UrlTest/Relay）
│       └── 启动/停止/状态
├── Tab 2: 节点代理（官方 v0.2.22 mihomo 单跳）
│   └── MihomoPage（官方新增组件）
│       ├── 连接 Clash-Verge
│       ├── 分组/节点浏览
│       └── 节点延迟测试
└── Tab 3: 远端代理（原有单跳 HTTP/SOCKS5）
```

### 3.4 create-window 代理选择 drawer 冲突消解

`create-window-proxy-drawer.tsx`（官方 +1664 行，变更量最大）需支持**三选一**：
- 远端代理（官方原有）
- 本地 Mihomo 代理（官方 v0.2.22 新增）
- **链式代理（本地 proxy_chain）** ← 本地新增选项

融合后该 drawer 的代理类型枚举：
```typescript
type ProxyType = 'remote' | 'mihomo-local' | 'chain'
```

### 3.5 proxy_chain 代码完整保留清单

**Rust 后端**（全部保留，作为独立模块）：
```
src-tauri/src/services/proxy_chain/
├── mod.rs              # 模块导出
├── service.rs          # ProxyChainService 门面（11 方法）
├── storage.rs          # 持久化 <config_dir>/proxy_chain/config.json
├── generator.rs        # 生成 Mihomo YAML（dialer-proxy 多跳）
├── process.rs          # mihomo 进程管理（RUNTIME 单例）
├── subscription.rs     # 订阅解析（Clash YAML + vmess/trojan/ss/vless/socks/http）
├── api.rs              # MihomoApiClient（external-controller）
├── redact.rs           # 脱敏/还原
└── types.rs            # ProxyChainConfig/ProxyNode/ProxyPolicy 等

src-tauri/src/commands/proxy_chain.rs   # 13 个 #[tauri::command]
```

**前端**（全部保留）：
```
plugins/pages/proxy-center/src/
├── index.tsx                       # ProxyCenterPage（改为 Tab 容器）
├── api/proxy-chain.ts              # 链式代理 API（616 行）
├── components/proxy-chain-panel.tsx # 链式面板主组件
└── ...（16 个代理 CRUD 组件 + 9 个 hooks）
```

### 3.6 mihomo 二进制依赖
proxy_chain 的 `process.rs::resolve_mihomo_binary` 依赖 `mihomo.exe`（bundled 或 `SIMPRINT_MIHOMO_PATH` 环境变量）。当前 `tauri.conf.json` 的 `bundle.resources` 仅含 `simprint-runtime.exe`，未见 mihomo 打包。

**处理**：部署/打包时另行分发 mihomo 二进制，或在 bundle.resources 中补充。
