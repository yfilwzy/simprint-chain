# 03 - 登录指向自建服务器

> 目标：登录与全部云端数据指向用户自有服务器，复用客户端原有 RSA+AES-GCM 加密协议

---

## 一、客户端改造面（极小）

用户选择"复用同套加密协议"，客户端代码层**零改动**，仅改配置文件的 4 组 URL。

### 1.1 改造点清单

| 改造点 | 文件 | 字段 | 原值（官方） | 新值（自建） |
|--------|------|------|-------------|-------------|
| 主 API 地址 | `config.*.toml` `[server]` | `base_url` | `https://api.simprint.app/api/` | `https://<YOUR_DOMAIN>/api/` |
| 应用密钥 | `config.*.toml` `[server]` | `secret_key` | `<官方原值已脱敏>` | `<YOUR_API_SECRET>` |

| runtime 清单 | `config.*.toml` `[updater]` | `runtime_latest_json_url` | `https://pub-...r2.dev/.../latest.json` | `https://<YOUR_DOMAIN>/update/simprint-runtime/latest.json` |
| webview 包 | `config.*.toml` `[webview]` | `downlaod_url` | `https://pub-...r2.dev/webview-fixed.zip` | `https://<YOUR_DOMAIN>/update/webview-fixed.zip` |

### 1.2 客户端代码层（全部不动）
以下文件零改动，因为加密协议不变：
- `infrastructure/http/encryption/aes.rs`（AES-256-GCM）
- `infrastructure/http/encryption/rsa.rs`（RSA-2048）
- `infrastructure/main_server/interceptors/request.rs`（加密 + 鉴权拦截器）
- `infrastructure/main_server/interceptors/crypto.rs`（响应解密）
- `infrastructure/persistence/credential/mod.rs`（token + 公钥管理）
- `services/auth/login.rs` / `register.rs`（登录注册逻辑）

### 1.3 配置加载机制（理解要点）
- `core/config/loader.rs` 编译期 `include_bytes!(OUT_DIR/config_encrypted.bin)` 嵌入加密配置
- `build.rs` 负责把 `config.*.toml` 加密生成 bin
- 改 config 后**必须重跑 build.rs** 重新生成 bin，否则配置不生效
- 运行时优先读本地 `config.development.toml`（dev feature），否则读嵌入 bin

## 二、加密协议完整规格（服务端必须实现）

### 2.1 协议概览
客户端与服务端之间使用 **RSA-2048 + AES-256-GCM 混合加密**：

```
┌──────────┐                    ┌──────────┐
│ 客户端    │                    │ 服务端    │
└────┬─────┘                    └────┬─────┘
     │                                │
     │  ① GET secret/public/key       │
     │ ─────────────────────────────> │  返回服务端 RSA 公钥(PEM, PKCS1)
     │ <───────────────────────────── │
     │                                │
     │  ② 每次启动生成临时客户端 RSA 密钥对（仅本会话）│
     │                                │
     │  ③ 请求加密：                    │
     │     - 随机生成 AES-256 密钥       │
     │     - AES-GCM 加密 body         │
     │     - 用服务端公钥 RSA 加密 AES 密钥│
     │     - 发送 {data, encrypted:true, key} │
     │ ─────────────────────────────> │
     │                                │  服务端用私钥 RSA 解密 → 得 AES 密钥
     │                                │  → AES-GCM 解密 body → 校验 api_secret
     │                                │
     │  ④ 响应加密：                    │
     │                                │  - 用客户端公钥 RSA 加密新 AES 密钥
     │                                │  - AES-GCM 加密响应 body
     │ <───────────────────────────── │  返回 {data, encrypted:true, key}
     │                                │
     │  客户端用私钥 RSA 解密 → AES 解密  │
```

### 2.2 请求加密详细规格（客户端 `encrypt_request_body`）

```rust
// 输入：明文 JSON body（已注入 api_secret）
// 输出：{data, encrypted:true, key}

fn encrypt_request_body(data: &Value, server_public_key: &str) -> Value {
    let aes_secret = AesSecret::new();   // 随机 AES-256 密钥
    let json_string = serde_json::to_string(data);
    let encrypted_data = aes_secret.encrypt(json_string.as_bytes());
    // encrypted_data = base64(nonce[12B] || ciphertext)

    let encrypted_key = RsaSecret::encrypt_with_public_key(
        aes_secret.get_key_as_base64().as_bytes(),
        server_public_key
    );
    // encrypted_key = base64(RSA_OAEP(aes_key_base64))

    json!({
        "data": encrypted_data,
        "encrypted": true,
        "key": encrypted_key
    })
}
```

**服务端解密对应逻辑**（Go 伪代码）：
```go
func decryptRequest(body EncryptedBody, serverPrivKey *rsa.PrivateKey) (plainJSON, error) {
    // 1. RSA 解密 AES 密钥
    aesKeyBase64, err := rsa.DecryptOAEP(serverPrivKey, body.Key)  // base64 解码后解密
    aesKey, _ := base64.Decode(aesKeyBase64)

    // 2. AES-GCM 解密 data
    raw, _ := base64.Decode(body.Data)
    nonce := raw[:12]
    ciphertext := raw[12:]
    plaintext, _ := aesgcm.Open(nil, nonce, ciphertext, aesKey)

    // 3. 解析 JSON，校验 api_secret
    var jsonBody map[string]interface{}
    json.Unmarshal(plaintext, &jsonBody)
    if jsonBody["api_secret"] != config.ApiSecret {
        return error("invalid api_secret")
    }
    return jsonBody
}
```

### 2.3 响应加密详细规格

服务端响应必须用**请求中携带的客户端公钥**加密。客户端公钥获取方式：
- 登录请求 payload 含 `public_secret_key` 字段（客户端 RSA 公钥）
- 非登录请求：客户端公钥在会话建立后由服务端缓存（关联 access_token）

**服务端响应加密**（Go 伪代码）：
```go
func encryptResponse(plainJSON, clientPublicKeyPEM) EncryptedBody {
    aesKey := randomAES256()
    encryptedData := aesgcm.Seal(plainJSON, aesKey)  // base64(nonce||ciphertext)

    aesKeyBase64 := base64(aesKey)
    encryptedKey := rsa.EncryptOAEP(clientPublicKeyPEM, aesKeyBase64)

    return EncryptedBody{Data: encryptedData, Encrypted: true, Key: encryptedKey}
}
```

### 2.4 AES-GCM 细节
- 算法：AES-256-GCM
- nonce：12 字节随机
- 密文格式：`base64(nonce[12B] || ciphertext || tag[16B])`（GCM tag 附在 ciphertext 末尾）
- 密钥传输：`base64(RSA_OAEP_sha1(base64(aes_key)))`

### 2.5 RSA 细节
- 算法：RSA-2048
- padding：PKCS1v15（注意：客户端 `rsa.rs` 用 PKCS1v15Encrypt，服务端必须对齐）
- 公钥格式：`-----BEGIN RSA PUBLIC KEY-----`（PKCS1，非 PKCS8）

⚠️ **服务端实现注意**：RSA padding 必须与客户端对齐。客户端 `RsaSecret::encrypt_with_public_key` 若用 OAEP，服务端用 OAEP；若用 PKCS1v15，服务端用 PKCS1v15。**实现前必须读 `infrastructure/http/encryption/rsa.rs` 确认 padding 类型**。

## 三、服务端必须实现的端点

### 3.1 认证端点

| 端点 | 方法 | 说明 | 请求体 | 响应 |
|------|------|------|--------|------|
| `/secret/public/key` | GET | 返回服务端 RSA 公钥 | 无（不加密） | `{public_key: "-----BEGIN RSA PUBLIC KEY-----..."}` |
| `/users/login` | POST | 登录 | `{login_type, email, password/refresh_token, public_secret_key, api_secret}` | `{access_token, refresh_token, is_first_login, user_info, user_extra}` |
| `/users/register` | POST | 注册 | `{email, password, referral_code?, public_secret_key, api_secret}` | 同 login |
| `/users/refresh-credentials` | POST | 刷新 token | `{refresh_token, api_secret}` | `{access_token, refresh_token}` |

**LoginType 枚举**：
```rust
enum LoginType {
    Basic { email, password },           // 账密登录
    RememberPassword { email, refresh_token },  // 记住密码自动登录
}
```

## 四、数据库 Schema

```sql
-- 用户表
CREATE TABLE users (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    email           VARCHAR(255) UNIQUE NOT NULL,
    password_hash   VARCHAR(255) NOT NULL,        -- bcrypt
    is_first_login  BOOLEAN DEFAULT TRUE,
    user_info       JSONB,                         -- 灵活扩展
    user_extra      JSONB,
    created_at      TIMESTAMPTZ DEFAULT NOW(),
    updated_at      TIMESTAMPTZ DEFAULT NOW()
);

-- 会话/令牌表
CREATE TABLE sessions (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id         UUID REFERENCES users(id) ON DELETE CASCADE,
    access_token    VARCHAR(512) NOT NULL,
    refresh_token   VARCHAR(512) NOT NULL,
    expires_at      TIMESTAMPTZ NOT NULL,
    created_at      TIMESTAMPTZ DEFAULT NOW()
);

-- 邀请码表
CREATE TABLE referral_codes (
    code            VARCHAR(64) PRIMARY KEY,
    referrer_id     UUID REFERENCES users(id),
    used_count      INT DEFAULT 0,
    max_uses        INT DEFAULT 100
);

-- 工作区资源（自托管业务数据）
CREATE TABLE environments (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id         UUID REFERENCES users(id) ON DELETE CASCADE,
    name            VARCHAR(255),
    config          JSONB,
    created_at      TIMESTAMPTZ DEFAULT NOW()
);
-- proxies / groups / tags / workspaces 同构
```

## 五、api_secret 机制

`api_secret` 是应用层鉴权凭证（非加解密密钥），在 `interceptors/request.rs::encrypt` 中注入到每个加密请求体的 JSON：

```rust
obj.insert("api_secret".to_string(), ctx.config.server.secret_key.clone());
```

**服务端校验**：解密请求 body 后，校验 `api_secret` 字段 == 配置值。不符则拒绝。这是"合法 Simprint 客户端"的身份凭证。

**生成建议**：服务端启动时随机生成 20 字符密钥，写入客户端 `config.*.toml` 的 `secret_key`。

## 六、服务端技术选型

| 组件 | 选型 | 理由 |
|------|------|------|
| 语言 | **Go** | 编译型单二进制、内存占用低、RSA+AES 标准库完备、契合 2核2G |
| 数据库 | **PostgreSQL** | JSONB 支持灵活 schema、官方 install-server.sh 暗示用 PG |
| 反代 | **Nginx** | TLS 终结、反代、静态资源 |
| TLS | **Let's Encrypt** | 免费自动化证书 |
