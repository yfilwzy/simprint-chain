# 01 - 融合基线对齐方案

> 目标：将本地二开版（基线约 v0.1.0/v0.2.9 前）与官方上游 v0.2.26 深度融合，消解三方合并冲突

---

## 一、版本基线勘误（关键前提）

经官方仓库实测，**官方从 v0.2.9 起就没有 `tauri-plugin-updater` 依赖**，且早已存在自研的 `src-tauri/src/bin/updater.rs`，路径与本地完全相同。

这推翻了一个普遍误解：本地"移除官方 updater 插件"并非从头自研，而是基于官方已有的自研 updater 做的二次魔改。本地基线对应的官方锚点是 **v0.2.9 之前某个未打 tag 的 commit**。

**基线对齐策略**：以官方 v0.2.26 为主干，把本地魔改作为补丁叠加，而非反向把官方变更合并进本地。

## 四、对齐操作流程

### Step 1：建立官方基线分支
```bash
# 在 simprint-chain 仓库
git remote add upstream https://github.com/Simprint/simprint.git
git fetch upstream
git checkout -b merge/v0.2.26 upstream/main   # 官方 v0.2.26 为主干
```

### Step 2：本地魔改作为补丁叠加
本地魔改三项，逐项移植到 merge 分支：

1. **updater**：diff 本地 `services/updater/` + `bin/updater.rs` vs 官方对应文件，采纳官方分层，移植本地特有逻辑（如自有更新服务器 URL 配置）
2. **proxy_chain**：本地 `services/proxy_chain/`（8 文件）+ `commands/proxy_chain.rs` + 前端 `plugins/pages/proxy-center/src/` 整目录移植，作为独立模块
3. **config 卫生**：`.gitignore` 排除规则、`config.example.toml` 占位符、`NOTICE`/`LICENSE` 合规文件

### Step 3：冲突消解优先级
```
官方主干逻辑 > 本地魔改逻辑（除三项明确保留项）
  └─ updater：官方分层优先，本地仅保留 URL 配置差异
  └─ proxy_chain：本地完整保留，不与官方 mihomo 混合
  └─ 其他：采纳官方
```

### Step 4：验证
- `cargo check` 通过
- `pnpm lint` 通过
- updater 更新流程冒烟测试
- proxy_chain 链式代理冒烟测试
- 官方新功能（扩展管理/mihomo单跳/RPA）冒烟测试
