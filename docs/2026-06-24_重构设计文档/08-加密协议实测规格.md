# 08 - 加密协议实测规格（联调判据）

> 本文档是**客户端加密实现的逐行实测**，作为服务端实现的唯一权威判据。
> 服务端任何与此不符的实现都会导致客户端解密失败。
> 数据来源：`src-tauri/src/infrastructure/http/encryption/rsa.rs` + `aes.rs` 逐行确认。

---

## 一、RSA 规格（rsa.rs 实测）

| 项 | 值 | 代码依据 |
|----|-----|---------|
| 算法 | RSA-2048 | rsa.rs:19 `let bits = 2048` |
| **padding** | **PKCS1v15**（非 OAEP） | rsa.rs:5,51,61,79 全部 `Pkcs1v15Encrypt` |
| 公钥格式 | PKCS1 PEM | rsa.rs:31 `to_pkcs1_pem`，:73 `from_pkcs1_pem` |
| 公钥 PEM 头 | `-----BEGIN RSA PUBLIC KEY-----` | PKCS1 格式特征 |
| base64 编码 | standard（非 urlsafe） | rsa.rs:66,84 `general_purpose::STANDARD` |

**服务端必须**：
- `GET /secret/public/key` 返回 PKCS1 PEM 公钥（`-----BEGIN RSA PUBLIC KEY-----` 开头）
- 解密请求 `key` 字段时用 RSA **PKCS1v15**（用 OAEP 必失败）
- 加密响应 `key` 字段时用客户端公钥 RSA **PKCS1v15**

## 二、AES 规格（aes.rs 实测）

| 项 | 值 | 代码依据 |
|----|-----|---------|
| 算法 | AES-256-GCM | aes.rs:1,9 `Aes256Gcm`，key 32 字节 |
| nonce | 12 字节随机 | aes.rs:10 `NONCE_SIZE = 12`，:50 `generate_nonce` |
| 密文格式 | **nonce(12B) 在前 + ciphertext 在后** | aes.rs:58-60 先 extend nonce 再 extend ciphertext |
| base64 编码 | standard | aes.rs:6 `STANDARD` |
| GCM tag | 附在 ciphertext 末尾 | aes-gcm crate 默认行为 |

**服务端必须**：
- 解密 `data` 字段：base64 解码 → 前 12 字节为 nonce → 其余为 ciphertext（含 tag）→ AES-GCM Open
- 加密响应 `data`：生成 nonce → AES-GCM Seal → nonce + ciphertext 拼接 → base64

## 三、密钥封装顺序（最高频踩坑点）

### 请求 `key` 字段（客户端→服务端）
```
key = base64( RSA_PKCS1v15_encrypt( base64(aes_key_bytes) ) )
```
**注意**：不是直接对 aes_key_bytes 做 RSA，而是**先对 aes_key_bytes 做 base64 编码成字符串**，再对该字符串的 UTF-8 字节做 RSA 加密，最后整体 base64。

代码依据（aes.rs:195-199）：
```rust
let encrypted_data = public_key.encrypt(&mut rng, rsa::Pkcs1v15Encrypt, key.as_bytes());
// key.as_bytes() 即 base64(aes_key) 的字节
let encrypted_key = base64::engine::general_purpose::STANDARD.encode(&encrypted_data);
```

### 响应 `key` 字段（服务端→客户端）
镜像操作：
```
key = base64( RSA_PKCS1v15_encrypt_with_client_pub( base64(new_aes_key_bytes) ) )
```

## 四、完整请求/响应载荷结构

```json
{
  "data": "<base64(nonce[12B] || ciphertext_with_tag)>",
  "encrypted": true,
  "key": "<base64(RSA_PKCS1v15(base64(aes_key)))>"
}
```

## 五、解密后的明文 JSON（请求体）

解密 `data` 得到的明文 JSON 含业务字段 + 鉴权字段：
- `api_secret`：应用层鉴权密钥，须 == 服务端 config（每个加密请求都注入）
- `public_secret_key`：客户端 RSA 公钥（PKCS1 PEM），服务端用它加密响应（登录请求必带）

## 六、服务端自检清单

服务端实现后，用以下检查项自测（任一不符则联调必失败）：

- [ ] RSA padding 是 PKCS1v15（不是 OAEP）
- [ ] 公钥返回 PKCS1 PEM（`BEGIN RSA PUBLIC KEY`，非 `BEGIN PUBLIC KEY`）
- [ ] AES 解密时 nonce 取前 12 字节
- [ ] `key` 字段先 base64(aes_key) 再 RSA（不是直接 RSA aes_key）
- [ ] base64 全用 standard encoding（非 urlsafe）
- [ ] 响应加密用客户端 public_secret_key（不是服务端自己的公钥）

## 七、padding 类型决策依据

rsa.rs 第 5 行明确 import `Pkcs1v15Encrypt`，第 51/61/79 行所有 encrypt/decrypt 调用都用此 padding。**这不是猜测，是代码逐行确认**。

03 设计文档 §2.4 曾写 `RSA_OAEP_sha1`，那是文档笔误，**以客户端实测 PKCS1v15 为准**。服务端若按 OAEP 实现必失败。
