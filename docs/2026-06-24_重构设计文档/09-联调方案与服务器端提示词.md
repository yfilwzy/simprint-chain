## 联调期间你执行的监控命令（收到联调请求时跑）

### 1. 实时日志（看客户端请求是否到达、解密是否成功）
```bash
docker logs -f --tail 100 simprint-server
```
重点关注：
- `login decrypt failed` → 加密协议不匹配（最严重，检查 padding/密钥）
- `invalid api_secret` → 客户端 config 的 secret_key 与服务端不符
- `db error` → 数据库问题
- 正常应看到 Login handler 无报错 + sessions 表新增记录

### 2. 查最近登录请求详情（解密后字段是否正确）
```bash
docker logs --tail 200 simprint-server 2>&1 | grep -iE "login|register|decrypt|api_secret|error"
```

### 3. 数据库状态（注册/登录是否落库）
```bash
docker exec simprint-postgres psql -U simprint_user -d simprint_db -c \
  "SELECT id, email, is_first_login, created_at FROM users ORDER BY created_at DESC LIMIT 5;"
docker exec simprint-postgres psql -U simprint_user -d simprint_db -c \
  "SELECT s.user_id, u.email, s.created_at FROM sessions s JOIN users u ON s.user_id=u.id ORDER BY s.created_at DESC LIMIT 5;"
```

### 4. Caddy 反代是否正常（4xx/5xx 检查）
```bash
tail -50 /var/log/caddy/simprint.log 2>/dev/null || journalctl -u caddy --since "10 min ago" --no-pager | tail -30
```

### 5. 端到端自检（公钥 + 健康检查 + DB）
```bash
curl -s https://api.yfilwzy.cc.cd/simprint/api/v1/secret/public/key | head -c 80
curl -s http://127.0.0.1:8090/health
docker exec simprint-postgres psql -U simprint_user -d simprint_db -c "SELECT count(*) FROM users;"
```

## 常见联调故障决策树（收到客户端报错时按此排查）

| 客户端症状 | 排查命令 | 可能原因 | 处置 |
|-----------|---------|---------|------|
| "连接服务器失败"/splashscreen 卡住 | curl 公钥端点 | 公钥拉不到 / TLS / Caddy | 确认 Caddy + 证书 |
| 登录"成功"但 token 存不进 | 看日志 login 是否报错 | 响应缺 {code,message,data} 外层 | 报告，由本地修服务端响应封装 |
| decrypt failed | `docker logs grep decrypt` | RSA padding 不匹配 / 客户端公钥格式 | 核对客户端 rsa.rs padding |
| invalid api_secret | grep api_secret 日志 | 客户端 secret_key ≠ 服务端 | 核对 config.production.toml |
| 401 unauthorized | grep sessions | access_token 过期/不存在 | 走 refresh-credentials |
| 业务请求无响应 | curl local-api | LocalApiProxy 路由/鉴权 | 看日志路径解析 |

## 处置原则

- **只观测、只诊断**：收到请求后跑监控命令，把日志/DB 状态报告给主控
- **不主动改代码**：服务端代码已稳定，任何修改须本地（客户端侧）确认是服务端 bug 后，由本地出 patch 再部署
- **紧急回滚**：若联调导致服务端崩溃，`docker compose -f /opt/simprint/docker-compose.yml restart server` 重启
- **加密相关任何报错立即报告**：这是协议层问题，必须本地介入判断

## 三、给本地的客户端联调清单（你自己执行）

服务器端已就绪，球在客户端。按以下步骤联调：

### 步骤 1：配置客户端 config.production.toml
```bash
cd C:\Users\ADMIN\simprint-build-src
# 用交付包里的配置（已填好 secret_key）
copy "D:\desktop\服务器部署\服务器端simprint\交付物\config.production.toml" "src-tauri\config.production.toml"
```
验证 secret_key = `s3eqVcjYIwxb5i3YtBP6`（与服务端一致）。

### 步骤 2：重跑 build.rs（关键，否则配置不生效）
客户端 config 是编译期嵌入加密的，改了 config 必须重跑 build.rs 重新生成 config_encrypted.bin：
```bash
cd src-tauri
cargo build  # 会触发 build.rs 重新加密配置
```

### 步骤 3：客户端开发模式联调
```bash
cd C:\Users\ADMIN\simprint-build-src
pnpm install      # 若没装依赖
pnpm tauri dev    # 开发模式启动（不需完整打包）
```
观察：
- splashscreen 启动 → 应能拉到服务器公钥（不再卡"连接服务器失败"）
- 进入登录页 → 注册新账号 → 登录

### 步骤 4：联调验证点
| 验证 | 预期 | 失败排查 |
|------|------|---------|
| 启动拉公钥 | splashscreen 不卡 | curl 公钥端点 + 看 config base_url |
| 注册账号 | 成功，DB users 表有记录 | 看 docker logs simprint-server |
| 账密登录 | 登录成功，token 存入 | 看日志 + 客户端是否报 decrypt failed |
| 业务请求 | 能创建环境/代理 | 看 local-api 路径 |
| 退出再登录 | remember 模式自动登录 | 看 refresh_token |

### 步骤 5：联调遇到问题时
- **服务器日志**（已配免密）：`ssh simprint-server "docker logs --tail 50 simprint-server"`
- **客户端日志**：RUST_LOG=debug 看加密/解密详情
- **协议问题**：对照 08-加密协议实测规格.md 逐项核
