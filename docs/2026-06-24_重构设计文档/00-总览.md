# Simprint-Chain 重构设计总览

> 文档版本：1.0 · 编制日期：2026-06-24 · 编制者：yfilwzy
> 仓库：https://github.com/yfilwzy/simprint-chain · 许可证：AGPLv3

---

## 一、项目目标

将本地二开版 Simprint-Chain（当前基线约 v0.1.0 源码状态）与官方上游最新版 **v0.2.26** 进行深度融合，达成四个核心目标：

1. **版本同步**：完整吸收官方 v0.2.9 → v0.2.26 的全部功能演进（49 commits / 160 文件变更）
2. **魔改保留**：完整保留本地三项魔改（自研更新器、proxy_chain 链式代理、config 卫生），零功能损失
3. **隐私自托管**：登录与全部云端数据指向用户自有服务器，复用客户端原有 RSA+AES-GCM 加密协议
4. **MCP 增强**：参考 CLI-Anything 设计模式，用纯 Rust 重构 MCP 工具注册表架构，提升工具暴露的规范性与可扩展性

## 二、范围与约束

### 技术栈
| 层 | 技术 | 版本 |
|----|------|------|
| 桌面运行时 | Tauri | 2.x |
| 后端逻辑 | Rust | edition 2024 |
| 前端框架 | React | 19 |
| 构建工具 | Vite + pnpm | 9 |
| MCP SDK | rmcp | 1.2.0 |
| 本地代理核心 | Mihomo | 外部二进制 |
| 自建服务器 | Go + PostgreSQL + Nginx | 待定 |

### 硬约束
- **AGPLv3 合规**：融合官方代码必须保留协议、标注衍生（NOTICE 文件已就绪）
- **隐私边界**：所有用户数据上传必须收敛到用户自有服务器，禁止任何第三方泄露点
- **资源约束**：服务器为 2核2G/40G/Ubuntu 24.04，部署方案须在此规格内可行
- **操作习惯**：前端排版逻辑、按键设计必须符合桌面操作习惯，设计逻辑深度复核
- **版本基线**：对齐官方 v0.2.26，本地衍生版本号 `v0.2.26-chain.1`

### 不在范围内
- 不引入 Python 运行时（MCP 增强纯 Rust 实现）
- 不改变桌面应用的核心技术栈（仍是 Tauri+React+Rust）
- 不实现第三方 GUI 软件的 MCP 驱动（仅增强 Simprint 自身能力暴露）

## 三、四大核心决策（已推演定稿）

以下决策经 sequential-thinking 深度推演，详见各专项文档。

### 决策 1：proxy_chain 保留为独立模块
- **结论**：采用方案 A——proxy_chain 作为独立功能完整保留，官方 v0.2.22 的 Mihomo 单跳代理前端整体采纳，两者在前端分层共存
- **依据**：本地 proxy_chain 是多跳链式（落地 SOCKS5 + dialer-proxy 指向前置组），与官方 Mihomo 单跳节点编排是不同概念。proxy_chain 自行生成完整 Mihomo YAML 并独立启动进程，不依赖官方 controller API，两者数据隔离无冲突
- **详情**：见 [02-魔改保留方案.md](./02-魔改保留方案.md)

### 决策 2：复用 RSA+AES-GCM 加密协议
- **结论**：客户端只改 4 个 URL 配置（base_url/secret_key/三个 updater URL），加密代码层零改动；服务端用 Go 完整实现协议栈
- **依据**：客户端加密逻辑（infrastructure/http/encryption + main_server/interceptors）与协议强绑定，改协议成本远高于复用。工作量转移至服务端
- **详情**：见 [03-登录指向自建服务器.md](./03-登录指向自建服务器.md)

### 决策 3：MCP 注册表架构重构
- **结论**：抽取 `McpTool` trait + `ToolRegistry` 注册表，借鉴 CLI-Anything 的 registry.json / SKILL.md / repl_skin 设计模式，纯 Rust 重写
- **依据**：CLI-Anything 全仓 2539 文件仅 1 个提及 MCP，其价值在设计模式而非代码。当前 MCP 的 `all_routes()` 硬编码聚合 7 模块，扩展性不足
- **详情**：见 [04-MCP增强设计.md](./04-MCP增强设计.md)

### 决策 4：Go + PostgreSQL + Nginx 部署
- **结论**：服务端用 Go（编译型单二进制、内存占用低、契合 2核2G），PostgreSQL 数据库，Nginx 反代 + Let's Encrypt TLS
- **依据**：2核2G 资源评估显示 Go+PG+Nginx 总内存约 1G，留 1G 缓冲；Go 的 RSA+AES 标准库完备，协议实现可靠
- **详情**：见 [05-服务器部署提示词.md](./05-服务器部署提示词.md)
